# ADR 003: Propshaft Tailwind Stub Filter

**Status**: Implemented
**Date**: 2026-03-03
**Context**: Kiso engine CSS not loading correctly in host apps using Rails 8.1 default `stylesheet_link_tag :app`

---

## The Problem

Host apps using the Rails 8.1 default `stylesheet_link_tag :app` get a 404
console error and potentially broken styles. The browser tries to fetch
Kiso's engine CSS via an absolute filesystem path like:

```
GET /Users/steve/.local/share/mise/installs/ruby/3.4.8/lib/ruby/gems/3.4.0/gems/kiso-0.2.0.pre/app/assets/tailwind/kiso/engine.css
→ 404 Not Found
```

This was discovered while integrating Kiso into a real host app (ApproveThis)
that used the standard Rails 8.1 generated layout with `stylesheet_link_tag :app`.

---

## Background: How Three Systems Interact

Three independent Rails systems are involved. The bug exists in the gap
between them — each one works correctly in isolation, but together they
produce an unintended result.

### 1. tailwindcss-rails: Engine CSS Bundling

`tailwindcss-rails` (v4.x) provides a mechanism for Rails engines to ship
Tailwind CSS. An engine places its CSS at a conventional path:

```
<engine>/app/assets/tailwind/<engine_name>/engine.css
```

For Kiso, this is:

```
kiso-0.2.0.pre/app/assets/tailwind/kiso/engine.css
```

This file contains Kiso's `@theme` tokens (semantic colors, fonts), `@source`
directives (so Tailwind scans Kiso's partials/themes for utility classes),
dark mode `.dark {}` overrides, `@import`s for component CSS (checkbox,
dashboard, input-otp, etc.), and `@layer base` body defaults.

The `tailwindcss:engines` Rake task (which runs automatically before
`tailwindcss:build` and `tailwindcss:watch`) scans all loaded Rails engines
for this conventional file. When found, it generates an intermediate "stub"
file in the **host app**:

```
<host>/app/assets/builds/tailwind/<engine_name>.css
```

For Kiso, this generates:

```
app/assets/builds/tailwind/kiso.css
```

The stub contains a single `@import` with an **absolute filesystem path**:

```css
/* DO NOT MODIFY THIS FILE, it was auto-generated by tailwindcss-rails */

@import "/Users/steve/.local/share/mise/installs/ruby/3.4.8/lib/ruby/gems/3.4.0/gems/kiso-0.2.0.pre/app/assets/tailwind/kiso/engine.css";
```

This absolute path is intentional — the Tailwind CLI can resolve filesystem
paths during compilation. The stub is a build-time intermediate: the Tailwind
CLI reads it, follows the `@import`, inlines the engine's CSS, and produces
a single compiled output file with everything merged.

The host app's source CSS references the stub via a relative import:

```css
/* app/assets/tailwind/application.css */
@import "tailwindcss";
@import "../builds/tailwind/kiso.css";
```

After compilation, the output at `app/assets/builds/tailwind.css` contains
all of Kiso's tokens, utility classes, and component CSS fully inlined. No
`@import` statements remain. This compiled file is what the browser should
receive.

**Source references:**
- Stub generation: `tailwindcss-rails` `lib/tailwindcss/engines.rb` `Tailwindcss::Engines.bundle`
- Installer logic: `lib/install/install_tailwindcss.rb` (skips adding `stylesheet_link_tag "tailwind"` if `:app` is present)
- Upgrade logic: `lib/install/upgrade_tailwindcss.rb` (removes `stylesheet_link_tag "tailwind"` if `:app` is present)
- Source exclusion: `lib/tailwindcss/engine.rb` initializer `"tailwindcss.exclude_asset_path"` (excludes `app/assets/tailwind` from Propshaft but NOT `app/assets/builds/tailwind`)

### 2. Propshaft: Asset Serving and the `:app` Symbol

Propshaft (v1.3.x) is the default asset pipeline in Rails 8. It serves
static assets from configured "load path" directories.

The load path is built from `app/assets/` subdirectories of the host app
and all loaded engines. For a typical host app, this includes:

```
app/assets/builds/        ← compiled CSS/JS output
app/assets/stylesheets/   ← traditional CSS
app/assets/images/        ← images
```

Propshaft walks each load path directory **recursively** to build its asset
index. So `app/assets/builds/` yields:

```
tailwind.css              ← logical path: "tailwind.css"
tailwind/kiso.css         ← logical path: "tailwind/kiso.css"
```

Both are servable assets. Requesting `/assets/tailwind-<digest>.css` returns
the compiled output. Requesting `/assets/tailwind/kiso-<digest>.css` returns
the raw stub.

Propshaft's `Helper` module overrides `stylesheet_link_tag` to support
two special symbols:

```ruby
# propshaft-1.3.1/lib/propshaft/helper.rb
def stylesheet_link_tag(*sources)
  case sources.first
  when :all
    sources = all_stylesheets_paths     # every CSS in the load path
  when :app
    sources = app_stylesheets_paths     # CSS files under app/assets/**/*.css
  end
  # ... generates one <link> tag per source
end

def app_stylesheets_paths
  Rails.application.assets.load_path.asset_paths_by_glob(
    "#{Rails.root.join("app/assets")}/**/*.css"
  )
end
```

The `:app` symbol globs **every `.css` file** under the host app's
`app/assets/` directory tree. This includes the compiled output AND the
intermediate stubs:

```ruby
["tailwind.css", "tailwind/kiso.css"]
```

Each entry becomes a separate `<link>` tag in the HTML.

### 3. Rails 8.1: Default Layout Convention

Rails 8.1 generates application layouts with `stylesheet_link_tag :app`
as the default. The `tailwindcss-rails` installer explicitly treats this
as the canonical pattern:

- **Install**: If `:app` is already in the layout, it does NOT add
  `stylesheet_link_tag "tailwind"` (it considers `:app` sufficient)
- **Upgrade**: If BOTH `:app` and `"tailwind"` are present, it REMOVES
  the `"tailwind"` tag

This means `tailwindcss-rails` expects `:app` to be the primary way CSS
is loaded in modern Rails apps.

---

## The Bug: Where the Systems Collide

When `:app` expands to `["tailwind.css", "tailwind/kiso.css"]`, the browser
receives two `<link>` tags:

1. **`tailwind.css`** — the compiled output with all CSS inlined. Correct.
2. **`tailwind/kiso.css`** — the raw intermediate stub. **Incorrect.**

The browser parses the stub and encounters:

```css
@import "/Users/steve/.local/.../kiso-0.2.0.pre/app/assets/tailwind/kiso/engine.css";
```

The browser interprets the absolute filesystem path as a URL path and makes
an HTTP request:

```
GET http://localhost:3000/Users/steve/.local/.../engine.css → 404
```

The compiled `tailwind.css` already contains everything, so the 404 is
"just" a console error in most cases. But it's confusing, looks broken,
and in some configurations could cause subtle issues if the browser blocks
rendering on the failed CSS import.

### Why `stylesheet_link_tag "tailwind"` Works

Explicitly naming `"tailwind"` serves only `tailwind.css` — the compiled
output. The stub is never sent to the browser. This is what Kiso's Lookbook
and dummy app originally used, which is why we never saw the bug until
testing with a real host app using the Rails 8.1 default.

### Why `excluded_paths` Can't Fix This

`tailwindcss-rails` excludes `app/assets/tailwind/` (the source directory)
from Propshaft:

```ruby
# tailwindcss-rails engine.rb
initializer "tailwindcss.exclude_asset_path" do
  config.assets.excluded_paths << Rails.root.join("app/assets/tailwind")
end
```

But it does NOT exclude `app/assets/builds/tailwind/` (the stubs directory).

We attempted to add a similar exclusion in Kiso's engine:

```ruby
config.assets.excluded_paths << Rails.root.join("app/assets/builds/tailwind")
```

This does not work. Propshaft's `excluded_paths` filters the **load path
directory list** — it removes exact entries from the `config.assets.paths`
array. The paths array contains `app/assets/builds` (the parent directory),
not `app/assets/builds/tailwind` (the subdirectory). Since there's no exact
match, nothing is excluded.

The relevant Propshaft code:

```ruby
# propshaft-1.3.1/lib/propshaft/railtie.rb
Rails::Engine.initializer "propshaft.append_assets_path" do |app|
  app.config.assets.paths.unshift(*paths["app/assets"].existent_directories)
  app.config.assets.paths = app.config.assets.paths.without(
    Array(app.config.assets.excluded_paths).collect(&:to_s)
  )
end
```

Propshaft walks load path directories recursively (`all_files_from_tree`
in `load_path.rb`) but has no mechanism to exclude individual files or
subdirectories within a load path.

---

## Our Fix: PropshaftTailwindStubFilter

Since we can't exclude files via configuration, we filter them at the
helper level. Kiso's engine prepends a module onto `Propshaft::Helper`
that overrides `app_stylesheets_paths` to reject engine stubs:

```ruby
# lib/kiso/propshaft_tailwind_stub_filter.rb
module Kiso
  module PropshaftTailwindStubFilter
    def app_stylesheets_paths
      super.reject { |path| path.start_with?("tailwind/") }
    end
  end
end
```

```ruby
# lib/kiso/engine.rb
initializer "kiso.filter_tailwind_stubs" do
  ActiveSupport.on_load(:action_view) do
    if defined?(Propshaft::Helper)
      Propshaft::Helper.prepend(Kiso::PropshaftTailwindStubFilter)
    end
  end
end
```

### Why This Is Safe

- **Engine stubs always have logical paths like `tailwind/<name>.css`** —
  the `tailwindcss:engines` task creates them in `app/assets/builds/tailwind/`.
  No legitimate user CSS would be placed there (it's auto-generated and
  gitignored).

- **The compiled output is `tailwind.css`** (no slash) — it passes through
  the filter untouched.

- **Other engine stubs are also filtered** — if a host app uses multiple
  engines (e.g., Kiso + Devise), all their stubs are filtered. This is
  correct because every stub has the same absolute-path `@import` problem.

- **Guarded with `defined?(Propshaft::Helper)`** — if the app uses
  Sprockets instead of Propshaft, the prepend is skipped.

- **Only affects `:app` resolution** — the `app_stylesheets_paths` method
  is only called when `stylesheet_link_tag :app` is used. Explicit
  `stylesheet_link_tag "tailwind"` bypasses this entirely.

### Verification

Confirmed in the dummy app with `bin/rails runner`:

```ruby
# Before filter (raw Propshaft glob):
["tailwind.css", "tailwind/kiso.css"]

# After filter (what :app resolves to):
["tailwind.css"]
```

The browser receives a single `<link>` tag. No console errors. All Kiso
CSS (tokens, component classes, dashboard layout, dark mode) present in
the compiled output.

## Is This a Bug in tailwindcss-rails?

We believe so, but we want to verify thoroughly before filing.

**The case for it being a bug:**

1. `tailwindcss-rails` explicitly treats `:app` as the canonical pattern
   (installer/upgrader logic confirms this)
2. `tailwindcss-rails` already excludes `app/assets/tailwind/` from
   Propshaft (so it understands that build intermediates shouldn't be served)
3. But it doesn't exclude `app/assets/builds/tailwind/` — the stubs it
   generates there are also build intermediates
4. The stubs contain absolute filesystem paths that are meaningless to
   browsers

**The case for "working as designed":**

- Maybe engine CSS bundling is newer than `:app` support and the
  interaction wasn't considered
- Maybe nobody has shipped a production Rails engine with Tailwind CSS
  at this scale before (most engines use View Components, not ERB partials
  with Tailwind utility classes)
- The fix would be trivial in tailwindcss-rails: add
  `excluded_paths << Rails.root.join("app/assets/builds/tailwind")` to
  the existing exclusion initializer. But this only works if Propshaft
  adds the `builds/tailwind` subdirectory as a separate load path entry,
  which it currently does not. So the real fix might need to be in
  Propshaft (supporting subdirectory exclusion) or in tailwindcss-rails
  (writing stubs outside `app/assets/`).

**Questions for deeper research:**

1. Are there other Rails engines shipping Tailwind CSS via the
   `tailwindcss:engines` mechanism? If so, do they have this problem?
2. Has anyone reported this interaction between `:app` and engine stubs?
3. Would `tailwindcss-rails` maintainers prefer to write stubs to a
   non-asset location (e.g., `tmp/tailwind/`) or add a Propshaft exclusion?
4. Could Propshaft support glob-based or path-prefix exclusions in
   `excluded_paths` (not just exact directory matches)?
5. Is there a simpler fix we're missing — perhaps a Propshaft configuration
   option or a different file naming convention that would prevent stubs
   from being served?

## How Other Engines Handle This

### turbo_material (v0.30.0)

turbo_material is the engine referenced in tailwindcss-rails PR #554 — its
maintainer @bopm contributed the engine bundling feature. It ships 17 Material
Design components via ERB partials + helpers (similar to Kiso's `kui()` pattern).

**CSS strategy: pre-built fallback.** The gem maintainer runs the Tailwind build
locally and commits the compiled output to `app/assets/dist/turbo_material/tailwind.css`.
Host apps import this pre-built CSS as a static asset. This side-steps the
Propshaft `:app` stub problem entirely because the pre-built file is a real CSS
file with no `@import` directives — it's served as-is.

The engine also ships a `app/assets/tailwind/turbo_material/engine.css` for
dynamic `@source` scanning (the same mechanism Kiso uses), but the pre-built
file is the primary integration path.

**Trade-off:** Host apps cannot customize component styles via Tailwind
utilities because the CSS is pre-compiled. They can only override via CSS custom
properties. This is a fundamental limitation — the pre-built approach trades
flexibility for simplicity. For Kiso, where the whole point is composable,
overridable Tailwind utility classes via `css_classes:` and `class_variants`,
this trade-off is unacceptable.

**Other observations:**
- Uses CDN pins for Material Components Web JS (`jsDelivr`) — the async load
  pattern Kiso deliberately avoids (race conditions with Stimulus)
- No dark mode system
- Two-tier CSS adds maintenance burden — maintainer must rebuild and commit CSS
  whenever components change
- No evidence of a workaround for the `:app` stub problem on the dynamic path

### Maquina Components

Maquina Components uses the same `tailwindcss-rails` engine bundling mechanism
(`app/assets/tailwind/maquina_components_engine/engine.css` with `@source`
directives). It has the same `:app` stub vulnerability as Kiso. No workaround
is implemented — likely because users haven't hit it yet or are using explicit
`stylesheet_link_tag "tailwind"`.

Maquina uses a CSS-driven variant approach (`data-component` + `data-variant`
attributes with CSS rules) rather than Ruby `class_variants`. Simpler but less
composable than Kiso's approach.

### Summary: Kiso's position

Kiso is the only engine we've found that:
1. Uses dynamic `@source` scanning (not pre-built CSS)
2. Has discovered and fixed the Propshaft `:app` stub problem
3. Ships a comprehensive dark mode system
4. Uses Ruby-level variant composition (`class_variants` + `tailwind_merge`)

The pre-built approach used by turbo_material is a valid alternative but
fundamentally limits host app customizability. Our dynamic approach with the
`PropshaftTailwindStubFilter` workaround preserves full Tailwind flexibility
while protecting host apps from the stub bug.
