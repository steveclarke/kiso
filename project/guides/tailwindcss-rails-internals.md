# Guide: tailwindcss-rails Internals

A reading guide for understanding how `tailwindcss-rails` (v4.4.0) bootstraps
Tailwind CSS in Rails apps. Written for someone fluent with Tailwind 4 CSS
but not with the Rails integration internals.

**Source code**: `/Users/steve/src/vendor/tailwindcss-rails`

**Estimated reading time**: 45 minutes

---

## What This Gem Actually Is

`tailwindcss-rails` is a **build orchestrator**, not a CSS library. It:

1. Bundles the Tailwind CLI binary (via the separate `tailwindcss-ruby` gem)
2. Provides Rake tasks to run the CLI with the right arguments
3. Hooks into the Rails asset pipeline so CSS is compiled before serving
4. Ships a Puma plugin for watch mode during development
5. Generates intermediate files so Rails engines can ship Tailwind CSS

The gem itself has very little code — about 250 lines across 8 files. Most
of the heavy lifting is done by the Tailwind CLI.

---

## High-Level Flow

```
Installation (one-time)
  bin/rails tailwindcss:install
  → creates app/assets/tailwind/application.css
  → adds stylesheet_link_tag to layout
  → sets up Procfile.dev for watch mode

Build time (every compile)
  bin/rails tailwindcss:build
  → tailwindcss:engines task discovers engine CSS, generates stubs
  → Tailwind CLI reads application.css, follows @imports, scans @source paths
  → outputs compiled CSS to app/assets/builds/tailwind.css

Development (watch mode)
  Puma plugin or Procfile.dev
  → same as build, but with -w flag for file watching
  → rebuilds on any source file change

Serving
  Propshaft serves app/assets/builds/tailwind.css to the browser
  → ← THIS IS WHERE OUR BUG LIVES (stubs also served via :app)
```

## Read These Files In This Order

### 1. Entry Point: `lib/tailwindcss-rails.rb` (8 lines)


Nothing interesting here, but it tells you what the gem consists of.

---

### 2. Engine Setup: `lib/tailwindcss/engine.rb` (19 lines)

The Rails engine that integrates the gem into host apps. Two initializers:

**Initializer 1** — disables Rails stylesheet generators (line 5):
```ruby
Rails.application.config.generators.stylesheets = false
```
Since Tailwind handles all CSS, Rails shouldn't create component `.css` files.

**Initializer 2** — Propshaft source exclusion (lines 9-13):
```ruby
initializer "tailwindcss.exclude_asset_path", before: "propshaft.append_assets_path" do
  if Rails.application.config.assets.excluded_paths
    Rails.application.config.assets.excluded_paths << Rails.root.join("app/assets/tailwind")
  end
end
```

This tells Propshaft: "don't serve files from `app/assets/tailwind/` — that's
the source directory for the Tailwind CLI, not for browsers."

**This is where the bug lives.** It excludes `app/assets/tailwind/` (source)
but NOT `app/assets/builds/tailwind/` (stubs). The `before:` ordering was
itself a bug fix from issue #477 — originally it said `after:` which meant
the exclusion ran too late.

### 3. Engine CSS Bundling: `lib/tailwindcss/engines.rb` (23 lines) — THE KEY FILE

This is the engine support feature from PR #554. It's tiny but has big
implications.

```ruby
module Tailwindcss
  module Engines
    def self.bundle
      # Create the stubs directory in the host app
      FileUtils.mkdir_p(Rails.root.join("app/assets/builds/tailwind"))

      # Find every loaded Rails engine that has a conventional CSS file
      Rails::Engine.subclasses.select do |engine|
        engine.root.join(
          "app/assets/tailwind/#{engine.engine_name}/engine.css"
        ).exist?
      end.each do |engine|
        # Write a stub file with an @import pointing to the engine's CSS
        file_path = Rails.root.join(
          "app/assets/builds/tailwind/#{engine.engine_name}.css"
        )
        FileUtils.rm(file_path) if File.exist?(file_path)
        template = <<~TEMPLATE
          /* DO NOT MODIFY THIS FILE, it was auto-generated by tailwindcss-rails */

          @import "#{engine.root.join("app/assets/tailwind/#{engine.engine_name}/engine.css")}";
        TEMPLATE
        File.open(file_path, 'w') { |file| file.puts template }
      end
    end
  end
end
```

**What happens step by step:**

1. Creates `app/assets/builds/tailwind/` in the host app
2. Iterates all loaded Rails engines (Kiso, Devise, etc.)
3. For each engine, checks if it has `app/assets/tailwind/<name>/engine.css`
4. If yes, writes a stub file at `app/assets/builds/tailwind/<name>.css`
5. The stub contains a single `@import` with an **absolute filesystem path**

**Example stub** (`app/assets/builds/tailwind/kiso.css`):
```css
/* DO NOT MODIFY THIS FILE, it was auto-generated by tailwindcss-rails */

@import "/Users/steve/.local/share/mise/installs/ruby/3.4.8/lib/ruby/gems/3.4.0/gems/kiso-0.2.0.pre/app/assets/tailwind/kiso/engine.css";
```

**Why absolute paths?** The Tailwind CLI runs on the filesystem and can
resolve absolute paths. During compilation, it reads this stub, follows the
`@import`, reads Kiso's `engine.css`, processes all its `@theme` blocks,
`@source` directives, and `@import`s, and inlines everything into the
compiled output. The stub is a bridge between the host app's build process
and the engine's CSS.

**Why this creates the Propshaft problem:** The stubs live in
`app/assets/builds/tailwind/`, which is inside `app/assets/builds/` — a
Propshaft load path. Propshaft serves every CSS file it finds. When
`stylesheet_link_tag :app` globs `app/assets/**/*.css`, it picks up the
stubs and serves them to the browser. The browser sees the `@import` with
a filesystem path, interprets it as a URL, and gets a 404.

**Critical constraint:** The directory name must match `engine.engine_name`.
For `Kiso::Engine`, that's `kiso`. If names don't match, the engine is
silently skipped — no warning, no error.

### 4. CLI Command Builder: `lib/tailwindcss/commands.rb` (42 lines)

Builds the command-line arguments for the Tailwind CLI:

```ruby
def self.compile_command(debug: false, **kwargs)
  command = [
    Tailwindcss::Ruby.executable(**kwargs),  # path to tailwindcss binary
    "-i", Rails.root.join("app/assets/tailwind/application.css").to_s,  # input
    "-o", Rails.root.join("app/assets/builds/tailwind.css").to_s,       # output
  ]
  command << "--minify" unless debug || rails_css_compressor?
  # ...
end
```

The input is always `app/assets/tailwind/application.css`. This is the file
where the host app's `@import "../builds/tailwind/kiso.css"` line lives.
The output is always `app/assets/builds/tailwind.css` — the compiled result
that Propshaft serves.

`watch_command` adds `-w` for file watching (development).

The actual CLI binary comes from `tailwindcss-ruby` — a separate gem that
downloads the platform-specific Tailwind executable.

### 5. Rake Tasks: `lib/tasks/build.rake` (43 lines) — ORCHESTRATION

This is where everything comes together:

```ruby
namespace :tailwindcss do
  desc "Build Tailwind CSS"
  task build: [:environment, :engines] do |_, args|
    command = Tailwindcss::Commands.compile_command(debug: args.extras.include?("debug"))
    system(*command) || raise("Tailwind build failed")
  end

  desc "Watch for changes and rebuild"
  task watch: [:environment, :engines] do |_, args|
    command = Tailwindcss::Commands.watch_command(always: args.extras.include?("always"))
    system(*command)
  rescue Interrupt
    # graceful exit on Ctrl+C
  end

  desc "Bundle engine CSS files"
  task engines: :environment do
    Tailwindcss::Engines.bundle
  end
end
```

**Key detail:** Both `build` and `watch` depend on `:engines`. This means
`Tailwindcss::Engines.bundle` runs before every compilation — stubs are
always regenerated.

**Asset pipeline integration** (bottom of file):
```ruby
Rake::Task["assets:precompile"].enhance(["tailwindcss:build"])
```
This ensures CSS is compiled before assets are digested for production.

### 6. Installation: `lib/install/install_tailwindcss.rb` (56 lines)

The template that runs on `bin/rails tailwindcss:install`. Key decisions:

**Lines 5-11 — The `:app` detection:**
```ruby
unless File.read(APPLICATION_LAYOUT_PATH).match?(/stylesheet_link_tag :app/)
  insert_into_file ..., '<%= stylesheet_link_tag "tailwind" ... %>'
end
```

If the layout already has `stylesheet_link_tag :app`, it does NOT add a
separate `"tailwind"` tag. It assumes `:app` will pick up `tailwind.css`
from the builds directory. This is correct for the compiled output — but it
also means `:app` picks up the stubs.

**Lines 35-38 — Creates the source CSS:**
```ruby
copy_file "#{__dir__}/application.css", TAILWIND_ASSET_PATH
```
This creates `app/assets/tailwind/application.css` with just
`@import "tailwindcss";`. The host app then manually adds engine imports
like `@import "../builds/tailwind/kiso.css";`.

---

### 7. Upgrade: `lib/install/upgrade_tailwindcss.rb` (70 lines)

Migrates v3 apps to v4. The relevant part for our bug:

**Lines 31-35 — Removes duplicate stylesheet tags:**
```ruby
if File.read(APPLICATION_LAYOUT_PATH).match?(/stylesheet_link_tag :app/) &&
   File.read(APPLICATION_LAYOUT_PATH).match?(/stylesheet_link_tag "tailwind"/)
  gsub_file APPLICATION_LAYOUT_PATH, /stylesheet_link_tag "tailwind".*/, ""
end
```

If both `:app` and `"tailwind"` exist, it removes `"tailwind"`. This
confirms `:app` is the canonical pattern — the gem actively ensures it's
the only stylesheet mechanism.

---

### 8. Puma Plugin: `lib/puma/plugin/tailwindcss.rb` (77 lines)

Integrates watch mode directly into `rails server`:

```ruby
Puma::Plugin.create do
  def start(launcher)
    in_background do
      IO.popen(['bin/rails', 'tailwindcss:watch'], 'r+') do |io|
        # monitor process
      end
    end
  end
end
```

Forks a child process that runs `tailwindcss:watch`. When Puma stops, the
watch process is killed. Alternative to using Procfile.dev with Foreman.

## The Stub Problem: A Visual Walkthrough

Here's exactly what happens when a host app uses `:app`:

```
1. tailwindcss:engines runs
   ↓
   Scans Rails::Engine.subclasses
   ↓
   Finds Kiso::Engine has app/assets/tailwind/kiso/engine.css
   ↓
   Writes app/assets/builds/tailwind/kiso.css:
     @import "/absolute/path/to/kiso/engine.css"

2. tailwindcss:build runs
   ↓
   Tailwind CLI reads app/assets/tailwind/application.css
     @import "tailwindcss";
     @import "../builds/tailwind/kiso.css";  ← follows this
   ↓
   CLI resolves the absolute @import in the stub
   ↓
   Inlines everything from engine.css: tokens, @source, component CSS
   ↓
   Outputs app/assets/builds/tailwind.css (complete, no @imports left)

3. Browser requests a page
   ↓
   Layout has: stylesheet_link_tag :app
   ↓
   Propshaft calls app_stylesheets_paths
   ↓
   Globs: Rails.root.join("app/assets")/**/*.css
   ↓
   Finds: ["tailwind.css", "tailwind/kiso.css"]
   ↓
   Generates TWO <link> tags
   ↓
   tailwind.css loads fine (compiled output)
   tailwind/kiso.css loads → browser sees @import "/Users/steve/..."
   ↓
   Browser: GET http://localhost:3000/Users/steve/... → 404

4. With Kiso's PropshaftTailwindStubFilter:
   ↓
   app_stylesheets_paths is intercepted by our prepend
   ↓
   super returns ["tailwind.css", "tailwind/kiso.css"]
   ↓
   .reject { |path| path.start_with?("tailwind/") }
   ↓
   Returns: ["tailwind.css"]
   ↓
   ONE <link> tag. No 404.
```

## What to Look At When Reading the Source

When you open these files, focus on:

1. **`engines.rb`** — Read every line. It's only 23 lines and it's the heart
   of the issue. Notice how `engine.root.join(...)` produces the absolute path.

2. **`engine.rb`** — Notice what's excluded (`app/assets/tailwind`) and
   what's NOT excluded (`app/assets/builds/tailwind`). This is the gap.

3. **`build.rake`** — Notice that `:engines` is a dependency of both `:build`
   and `:watch`. Stubs are regenerated every time.

4. **`install_tailwindcss.rb`** — Notice the `unless` guard on line 6 that
   treats `:app` as sufficient. This is the gem saying "if you have `:app`,
   you don't need explicit `"tailwind"`."

5. **`upgrade_tailwindcss.rb`** — Notice lines 31-35 where it removes
   `"tailwind"` if `:app` is present. This is the gem saying "`:app` is
   canonical, remove the redundancy."

## Questions This Raises

1. Should stubs be written outside `app/assets/`? (e.g., `tmp/tailwind/`)
   The CLI can read from anywhere — the stubs don't need to be in the asset
   pipeline.

2. Should `tailwindcss-rails` add a Propshaft exclusion for
   `app/assets/builds/tailwind/`? It already does for `app/assets/tailwind/`
   — same pattern, different directory. But Propshaft's `excluded_paths`
   can't exclude subdirectories within a load path (only top-level entries).

3. Should `tailwindcss-rails` filter `app_stylesheets_paths` itself?
   It's their stubs, their integration — arguably they should own the fix.

4. Is there a Propshaft enhancement needed? Supporting subdirectory or
   glob-based exclusions in `excluded_paths` would make the ecosystem more
   robust.
